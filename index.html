Tissuestack

Installation Guide

Tissuestack requires linux as a running platform. Binary packages for the following distros exist: Ubuntu, Debian, CentOS, Fedora and OpenSUSE.

Below you'll find a more detailed description on how to install the package for any of the respective distros (incl. download links).

REMARKS:
For the more recent versions of Tissuestack you'll find two packages, one for each, the server as well as the client components:
tissuestack-VERSION.xxx (server) and tissuestack-clients-VERSION.xxx (client tools)

Any dependencies for the packages are dealt with by the package manager/install with only one, deliberate, exception:
The tissuestack-clients package contains a command line tool which relies on a java runtime.
Should a runtime version greater than 1.6 not be present on the system it has to be installed separately.

It's highly recommended that you pick the Tissuestack version that's the most recent (highest version number)!

Super/root user privileges are needed for installation!


Post Installation Notes

After successful installation the web interface should be accessible via the apache webserver at port 80.
Using any browser of your choice and entering the following replaced with the respective machine name or ip you should now be able to use Tissuestack:
http://machine_name_or_ip/

While Tissuestack does -as of now- not have a user concept, an admin password protects certain functionality such as upload/ import and deletion of datasets and configuration changes.
It defaults to @minTischYu and should be changed after installation.

It's worth noting that -for the client tools only- in order for PATH inclusions to take place, a new terminal session has to be established.
Once that has been done the shell completion should present you with all the Tissuestack client commands available.


Potential Issues

Since Tissuestack uses various system components and other software there is a potential that things can fail after what seems to have been a successful install on the surface.

That said it rarely happens especially not if a physical machine/VM is dedicated to Tissuestack.

The biggest problem in the context of installation exists on machines that have already a web server running that is not apache.
Here there will be an "argument" over usage of port 80 which needs to be addressed.
This can be done so by changing the configuration of any of the 2 web servers to an alternate port that's usually not blocked by the firewall, 8080 or 8008 for example.

Similarly, if there exist one or more site configurations for a apache configuration so they can get into each others' way.
This problem would have to be solved manually by the admin, tweaking configs/paths such that there is no conflict any more.

Other potential pitfalls include the firewall and/or selinux but ordinarily the common cases have been tested/accounted for and the install handles them appropriately.


CentOS

Version 6 and 7 of CentOS are supported. The installation is identical with the only exception of the download links.

For el6 download a tissuestack package/version from: tissuestack.org or tissuestack.com

Download the following third party dependencies:
DCMTK 3.6.0
MINC 2.1.0
NIFTI CLIB 2.0.0

For el7s download a tissuestack package/version from: tissuestack.org or tissuestack.com

Download the following third party dependencies:
DCMTK 3.6.0
MINC 2.1.0
NIFTI CLIB 2.0.0

Also install the epel repositories (if not already enabled), an easy way to do so is to use yum:

 yum install epel-release

To complete the actual installation just append all downloaded packages to the yum install call, separating them by spaces:
 yum install tissuestack-VERSION.rpm dep1 dep2 ...


Debian

The package has been built/tested using version 7

Download a tissuestack package/version from: tissuestack.org or tissuestack.com

To complete the actual installation there are two ways to proceed, the easiest/recommended one: to install gdebi which thereafter takes care of installation (incl. dependencies)...

 apt-get install gdebi
 gdebi tissuestack_VERSION.deb

Alternatively you can run the following series of dpkg and apt-get commands to perform the installation. You shouldn't be too concerned about any error messages on the way as long as the last step completes without an error.

 apt-get install apache2 postgresql
 dpkg -i tissuestack_VERSION.deb
 apt-get install -f

Note: the three apt-get calls mentioned could be replaced by aptitude calls.


Ubuntu

Download a tissuestack package/version from: tissuestack.org or tissuestack.com

Tissuestack Commandline Tool

Download TS_CL_Tool.jar

What can I do with CL tool?

You can connect to your Tissue Stack servers from a remote machine, query, modify, delete and import information. For a list of options available please have a look at the options.

Where can I get it?

You can use the download link above (.zip folder symbol) or use the TS client packages for linux.

Are there any pre-requisites to using the CL tool?

You need to have a physical connection from your machine to the remote server instance. The protocol used is http so that it should be sufficient to have port 80 open which is ordinarily the case.

After a download and following the instructions for running the tool below you should be good to go. Some options will require the admin password for the TS instance:
Do I need the admin password for the TS instance?

How can I run the CL tool?

You'll need to have a java runtime installed on your machine (>=1.6).
After downloading the package, you can place it whereever you like and execute it like this:

java -jar TS_CL_Tool.jar

Since this makes for a kind of ugly/inconvenient calling, it's better to wrap the call in a shell script and add the location to your system's default lookup path. This gives you the benefit of being able to choose an alias name and call from any location.

Here is an example of such a file for linux and bash: ts_cl
Simply replace $SOME_DIRECTORY with the actual directory that holds the jar file.

What options does the CL tool give me?

Running the CL tool without further argumens will provide you with some help.

./ts_cl
Output:
Available options:
--v <== verbose
--ts-conf file_location <== uses config file to get connect/session info
--login password <== logs you in returning a session
--password old new <== changes the present admin password
--list <== lists all datasets, for details use --query
--query id [detailed (value: true/false)] <== lists details for an individual dataset
--import filename <== imports dataset
--delete id [delete_file (value: true/false)] <== deletes dataset (configuration and optionally file)
--modify dataset_id column value <== modifies dataset table
--file delete file <== deletes a file on the server
--file rename old_location new_location <== renames a file on the server
--upload-dir [raw-files-only (value: true/false)] <== lists contents of upload directory
--tile start dataset_id [dimensions (x,y,z)] [zooms_levels (0,1,2)] <== start tiling
--tile list [status (active|done|error|canceled)] <== lists tiling tasks
--tile status task_id <== displays status information on tiling task
--tile cancel task_id <== cancel a tiling task
--convert start file_to_be_converted raw_file_name <== start conversion
--convert list [status (active|done|error|canceled)] <== lists conversion tasks
--convert status task_id <== displays status information on conversion task
--convert cancel task_id <== cancel a conversion task
--config list <== lists configuration table contents
--config query key <== displays a given configuration table value
--config change key value <== changes value for a configuration table entry

IMPORTANT:

Options HAVE be prefixed by a dash or double dash
They MAY have no further arguments or required/optional ones.
These arguments MUST be separated by space only and not be prefixed
String arguments with spaces and special characters that are used in shell environments such as [ or ~ (to name 2 only) HAVE to be escaped/quoted!

For example, an unquoted change of a dataset description
./ts_cl --modify 2 description mouse blockface

Output:
ID: 2
FILENAME: /opt/tissuestack/data/ambmc-c57bl6-model-symmet_v0.8.raw/br> DESCRIPTION: mouse/br>

whereas
./ts_cl --modify 2 description "mouse blockface"
will result in:
Output:
ID: 2
FILENAME: /opt/tissuestack/data/ambmc-c57bl6-model-symmet_v0.8.raw
DESCRIPTION: mouse blockface

The upcoming sections are going to address the available options individiually depending on the task that needs to be performed.

Do I need the admin password for the TS instance?

For some options e.g. dataset delete and import you'll need to supply the admin password for the server instance in question. There should, however, not be a need to, explicitly, execute the --login option (unless you wish to do so) since you are prompted when login is required.

After successful login a session token is returned by the server which is remembered in the respective (default) config file (see --ts-conf) you have been using. It will be valid until expiry upon which you'll be asked to provide the admin password again.

Connecting to a TS server instance

The best way to connect to a specific TS server is to specify a configuration file with the --ts-conf option. This has the advantage that you can have multiple configurations ready to be used.

If you don't specify a configuration file, the tool will look for a .tissuestack.conf in your user's home directory.

You can either create this file the first time or have it created for you when you connect to a TS instance the first time.

The --server option will provide a means to both override any configuration provided AND setting a non-existing one like the default. Note how in the example below --server is not given with successive calls!

./ts_cl --server http://tissuestack.org --list
Output:
ID: 2
FILENAME: /opt/tissuestack/data/ambmc-c57bl6-model-symmet_v0.8.raw
DESCRIPTION: ambmc c57bl6 symmetric model 0.8

./ts_cl --server http://tissuestack.org --list
Output:
ID: 2
FILENAME: /opt/tissuestack/data/ambmc-c57bl6-model-symmet_v0.8.raw
DESCRIPTION: ambmc c57bl6 symmetric model 0.8

What does a CL tool configuration file look like?

Given what you know about --ts-conf you can create and use as many server configuration files as you like.

Here are the contents of one such file that has 2 lines with only the first one being required really. The second one is filled in for you after login:

Contents:
tissuestack.instance=http://tissuestack.org
tissuestack.last_session_token=85a9ff52433011e69a9dfa163e38b228

How to import a dataset

use: --import file
./ts_cl --import /opt/tissuestack/upload/test.raw

Output:
IMPORTED: /opt/tissuestack/data/test.raw
ID: 25

How to delete a dataset

use: --delete id [delete_file (value: true/false)]
./ts_cl --delete 25 true

Output:
dataset deleted

How to modify a dataset's configuration?

Note that only a very limited number of database values can be altered. Ids and file names cannot for instance, neither can dataset dimensions. What can be altered are the following list that is also included in the query view as well:

  • is_tiled
  • zoom_levels
  • one_to_one_zoom_level
  • description
  • resolution_mm

use: --modify id column value
./ts_cl --modify 2 zoom_levels
"[0.250000,0.500000,0.750000,1.000000,1.250000,1.500000,1.750000,2.000000,2.250000,2.500000]"

Output:
modified dataset successfully

How to delete a file on the server?

IMPORTANT: Do not use this method to delete imported datasets but instead: --delete

use: --file delete file
./ts_cl --file delete /tmp/wrongly_uploaded.mnc

Output:
file deleted

How to rename a file on the server?

IMPORTANT: Do not use this method to rename imported datasets!

use: --file rename old_file new_file
./ts_cl --file rename /tmp/wrongly_named.mnc /tmp/rightly_named.mnc

Output:
file renamed

How to list files in the upload directory of the server?

use: --upload-dir [raw-files-only (value: true/false)]
./ts_cl --upload-dir true

Output:
FILE: something.raw

How to tile a dataset?

use: --tile start id [dimensions (x,y,z)] [zooms_levels (0,1,2)]
./ts_cl --tile start 2

Output:
Tiling started. Task id: 1467787217144615

How to list tiling tasks?

use: --tile list [status (active|done|error|canceled)]
./ts_cl --tile list active

Output:

STATUS: running

How to list the configuration table contents of a TS server?

The configuration table of a TS instance holds key/value entries used for running TissueStack. The following is a non-exhaustive list of entries that the command may output:

  • session_timeout_minutes
  • data_directory
  • default_drawing_interval
  • version
  • server_tile_directory
  • upload_directory
  • default_zoom_levels

use: --config list
./ts_cl --config list

Output:
KEY: version
VALUE: 2.1
DESCRIPTION: version
KEY: max_upload_size
VALUE: 107374182400
DESCRIPTION: the maximum number of bytes allowed to upload in one go: 100 gig by default
KEY: server_proxy_path
VALUE: server
DESCRIPTION: server proxy path (relative to the application's web root directory)
KEY: default_zoom_levels
VALUE: [0.25, 0.5, 0.75, 1.0, 1.25, 1.5, 1.75, 2.0, 2.25, 2.5]
DESCRIPTION: default zoom levels

How to query the TS server configuration table by key?

use: --config query key
./ts_cl --config query default_zoom_levels

Output:
KEY: default_zoom_levels
VALUE: [0.25, 0.5, 0.75, 1.0, 1.25, 1.5, 1.75, 2.0, 2.25, 2.5]
DESCRIPTION: default zoom levels

How to change a TS server configuration entry?

IMPORTANT: use with caution and only if you know what you are doing!

use: --config change key value
./ts_cl --config change default_zoom_levels
"[0.25, 0.5, 0.75, 1.0, 1.25, 1.5, 1.75, 2.0, 2.25, 2.5]"

Output:
KEY: default_zoom_levels
VALUE: [0.25, 0.5, 0.75, 1.0, 1.25, 1.5, 1.75, 2.0, 2.25, 2.5]
DESCRIPTION: default zoom levels
